# Archive Governance

## このアーカイブの目的

- Works Finder は、掲載中の作品を検索・絞り込み・比較しやすい形で見せるためのアーカイブです。
- 作品数を増やすこと自体よりも、一覧で見たときの比較可能性、説明可能性、更新時の再現性を優先します。
- 今回の運用ルールは、既存の 17 件を大きく動かさず、今後の追加・差し替え・見直しで判断がぶれない状態を作るためのものです。

## 掲載する作品の条件

- 一覧内での役割が明確で、他作品との差分を説明できること。
- `genre`、`siteType`、`purpose`、`features` などの既存軸で無理なく整理できること。
- `summary`、`challenge`、`designTone` を誇張せず、制作意図や判断理由として説明できること。
- サムネイルが一覧内で破綻せず、既存カードと並べても極端に浮かないこと。
- 公開物ではなくても掲載可能だが、その場合は実在クライアントの公開実績のように誤認されない文脈を守ること。
- `detailUrl` を設定する場合は、実際に公開済みで到達可能な詳細ページがあること。

## 見送る作品の条件

- 一覧に入れても既存作品との役割差が弱く、追加理由を説明しにくいもの。
- 内容が未整理で、`summary` や `challenge` を保守的な文言で説明できないもの。
- サムネイル候補が不安定で、一覧内で明度・色温度・情報密度のバランスを崩すもの。
- 実在実績のように見えすぎる一方で、公開可否や表現根拠が確認できていないもの。
- 将来的に差し替える前提が強く、現時点ではアーカイブ品質を下げるだけのもの。

## featured の考え方

- `isFeatured` は「目立たせたいもの全部」に付けるフラグではありません。
- 優先するのは、役割が違う代表作です。
- 例:
  クリニックLP、BtoBサービス、採用、アーカイブ、見積/プロダクト系のように、一覧全体の比較軸が広がる代表作を優先します。
- 同ジャンル・同目的・同トーンの作品が続く場合は、先に入っている featured を見直してから追加判断します。
- 今後の運用では、featured は全体で 6〜8 件程度を目安に再整理できる状態を保ちます。
- 今回のフェーズでは既存データを大きく動かさず、まずルールを固定し、次回以降の見直しで整理します。

## サムネイル選定・差し替えルール

- 明度レンジ:
  極端な白飛び、極端な黒つぶれは避け、既存カード群のダークトーンの中で輪郭が読めるレンジを優先します。
- トリミング:
  主要被写体やレイアウトの意図が切れないことを優先し、中央だけを漫然と抜くのではなく、カード比率で意味が残る切り方にします。
- 情報密度:
  一覧カードサイズで読めない細部や UI 部品が過密に詰まる画像は避け、ひと目で構図や空気感が伝わる密度に絞ります。
- 色温度:
  1 枚だけ極端に寒色・暖色へ寄せすぎず、近接表示したときに一覧全体の温度感を壊さないものを選びます。
- 文字入り画像の扱い:
  可読できない小さな文字が大量に入る画像は避け、文字が入る場合でも見出しやブランド名など意味のある要素に限定します。
- 一覧内で浮かないこと:
  単体で映えるよりも、一覧に並んだときの整合性を優先します。コントラスト、彩度、余白感が突出するものは再検討します。
- 差し替え判断:
  新サムネイルが単体で良くても、一覧全体の視線誘導を壊すなら採用しません。差し替え時は近接する数枚とセットで見比べます。

## works.js に新規追加するときの入力チェック項目

- `id` が一意で、既存ルールに沿った連番形式になっていること。
- `slug` が一意で、画像ファイル名と一致していること。
- `createThumbnailAssets(slug)` と実ファイルの組み合わせが崩れていないこと。
- `genre`、`siteType`、`purpose`、`features` が既存語彙と大きく乖離していないこと。
- `summary` はカードで読める長さに収め、誇張表現より説明可能性を優先していること。
- `challenge` は「何が難しかったか」「何を整えたか」が一文でわかること。
- `designTone` は比較軸として機能する範囲に留め、曖昧なキャッチコピー化を避けること。
- `pageCount`、`scale`、`budgetRange`、`durationRange`、各 boolean が内部整合していること。
- `detailUrl` は公開済み詳細ページがある場合のみ設定し、それ以外は `null` にすること。
- `field order` は既存オブジェクトの並びを大きく崩さないこと。
- 追加前に「この作品を入れる理由」と「featured 候補かどうか」を別々に確認すること。

## 更新時チェックリスト

- 作品数の増減が意図どおりか確認する。
- `id`、`slug`、画像参照、`thumbnailFallback` が壊れていないか確認する。
- `summary`、`challenge`、`designTone` が説明可能な文言になっているか確認する。
- `detailUrl` を追加した場合、実際に公開済みで到達可能か確認する。
- `isFeatured` を増やした場合、一覧全体で役割分担が保てているか確認する。
- サムネイル差し替え時は、単体ではなく一覧で並べて明度・色温度・情報密度を確認する。
- 検索、絞り込み、並び替え、比較、モーダル、URL 同期、localStorage、相談導線が崩れていないか確認する。
- 公開前に console error、画像 404、相対パス切れがないか確認する。
